## 1. 배경 및 목적

### 1.1 배경

- 개별 종목에 변동성완화장치(VI)가 발동되면 KRX·NXT 양 시장 모두 2분간 단일가매매로 전환된다.
- 단일가 종료 시 **NXT는 정시에 즉시 연속거래를 재개**하는 반면, **KRX는 랜덤엔드(0~30초 무작위 지연) 후 재개**한다.
- 이 재개 시점 불일치 구간에서 NXT에 잔존하는 매도호가와 KRX 예상체결가 사이에 일시적 가격 괴리가 발생할 수 있다 (예: 2026년 삼성전자 사례 — NXT 평균 299,745원 체결 vs KRX 단일가 302,000원, 약 2초 간격).

### 1.2 목적

- 실거래 진입 전에 **이 기회가 실제로 얼마나 자주, 어떤 크기로, 어떤 잔량과 함께 발생하는지** 정량 데이터를 수집한다.
- 수집 데이터로 진입 임계치, 기대 손익, 레그 리스크를 백테스트할 수 있는 기반을 만든다.

### 1.3 범위 제외 (Out of Scope)

- 실제 주문 전송 (Phase 1에서 주문 API 호출 금지 — 코드 수준에서 차단할 것)
- 시장가/지정가 체결 보장, 잔량 점유 경쟁 모델링
- KRX·NXT 외 거래소(ATS) 지원

## 2. 시스템 개요

### 2.1 아키텍처

```
┌─────────────────────────────────────────────────────┐
│                    Observer Process                  │
│                                                      │
│  [REST] oauth2/token ──► Access Token (자동 갱신)     │
│                                                      │
│  [WS] wss://api.kiwoom.com:10000/api/dostk/websocket │
│   ├─ 상시 구독: 1h (VI발동/해제, 전종목)                │
│   └─ 동적 구독 (VI 발동 종목별, TTL 관리):              │
│       ├─ {code}_NX : 0B 체결, 0D 호가잔량  (NXT)      │
│       └─ {code}    : 예상체결, 0B 체결     (KRX)      │
│                                                      │
│  [Engine] 스프레드 계산 → 비용 차감 → 기회 판정          │
│  [Sink]   CSV/SQLite 기록 + 구조화 로그                │
└─────────────────────────────────────────────────────┘
```

### 2.2 상태 머신 (종목 단위)

```
IDLE ──(1h VI발동 수신)──► WATCHING
WATCHING ──(KRX 첫 체결 수신)──► RESUMED
WATCHING/RESUMED ──(TTL 만료: 발동+210s)──► UNSUBSCRIBE ──► IDLE
```

## 3. 기능 요구사항 (FR)

### FR-01 인증

- `POST {host}/oauth2/token`에 `grant_type=client_credentials`, `appkey`, `secretkey`로 토큰을 발급한다.
- 토큰은 메모리에만 보관하며 파일로 저장하지 않는다.
- WebSocket 재연결 누적 실패(연속 3회 이상) 시 토큰을 재발급한다.
- 운영/모의 서버를 환경변수 또는 설정값(`USE_MOCK`)으로 전환 가능해야 한다.
  - 운영: `api.kiwoom.com` / 모의: `mockapi.kiwoom.com`

### FR-02 WebSocket 세션 관리

- 접속 후 `{"trnm":"LOGIN","token":...}` 전송, `return_code=0` 확인 후에만 구독을 시작한다.
- 서버 `PING` 프레임은 수신 원문 그대로 echo 응답한다.
- 비정상 종료 시 지수 백오프(1s→2s→…→최대 30s)로 자동 재연결하고, 재연결 성공 시 **상시 구독(1h)과 활성 동적 구독 전체를 복원**한다.

### FR-03 VI 이벤트 감지

- 실시간 타입 `1h`(VI발동/해제)를 상시 구독한다.
- 전종목 수신이 불가한 경우를 대비해, 감시 유니버스(예: KOSPI200 + 거래대금 상위 N)를 설정 파일로 주입 가능해야 한다.
- VI 이벤트 수신 시 원문(raw values)을 `vi_events_*.csv`에 즉시 기록한다 (발동/해제 구분, 발동가격, 정적/동적 구분 FID 포함).
- 동일 종목 중복 발동은 기존 WatchState를 갱신(TTL 연장)한다.

### FR-04 동적 시세 구독

- VI 발동 종목 `code`에 대해 다음을 등록한다:
  - `{code}_NX` → `0B`(체결), `0D`(호가잔량)
  - `{code}` → 예상체결 타입, `0B`(체결; KRX 재개 감지용)
- 등록/해제는 `REG`/`REMOVE` 프레임을 사용하며, 응답 `return_code≠0`이면 1회 재시도 후 경고 로그를 남긴다.
- TTL(기본 210초 = 단일가 120s + 랜덤엔드 30s + 여유 60s) 경과 시 자동 해제한다.

### FR-05 스프레드 계산 및 기회 판정

- 양 시장 데이터가 모두 존재할 때마다(틱 단위) 다음을 계산·기록한다:
  - `spread = KRX예상체결가 − NXT최우선매도호가`
  - `cost = KRX예상체결가 × TAX + (매수가+매도가) × FEE`
  - `net = spread − cost`
  - `opportunity = (net > cost × (EDGE_BUFFER − 1))`
- 비용 파라미터는 설정값으로 외부화한다 (기본: `TAX=0.0015`, `FEE=0.00015/편도`, `EDGE_BUFFER=1.5`).
- NXT 최우선 매도호가의 **잔량**이 수신되는 경우 함께 기록한다 (실현 가능 수량 분석용).

### FR-06 KRX 재개(랜덤엔드) 감지

- 동적 구독 중 KRX `0B` 첫 체결 수신 시각을 랜덤엔드 시점으로 기록한다 (`KRX_RESUME` 이벤트).
- VI 발동 → KRX 재개까지의 경과 시간(랜덤엔드 분포)을 사후 분석할 수 있어야 한다.

### FR-07 데이터 적재 (✅ SQLite 시계열, 백테스트용)

- **`backend/services/vi_arb_store.py`** — hub 방송 메시지를 **`data/vi_arb.db`(SQLite)** 에 시계열로 영속화.
- 모든 행에 `ts`(ms 정밀도 KST ISO8601) + `source`(mock|kiwoom) 태그 → 백테스트는 `source='kiwoom'` 필터.
- 테이블:
  - **`spreads`** (백테스트 코어, VI 단일가 틱 시계열): `ts, code, name, direction, sec_since_vi, krx_expected, nxt_best_ask, nxt_ask_qty, spread, cost, net_spread, net_pct, max_profit_pct, first_buy, opportunity`
  - **`vi_events`**: `ts, code, name, kind(발동/해제), gubun(정적/동적), direction, vi_pct, trigger_price`
  - **`krx_resumes`**: `ts, code, name, randomend_sec`
  - **`orders` / `order_fills`**: 모의 주문·체결
- 인덱스: `(code, ts)`, `opportunity` → 종목별 시간범위·기회 쿼리.
- 시세 부호/콤마는 `_num()` 으로 정규화. 적재 현황: `GET /api/vi-arb/stats`.
- 백테스트 재구성 예: `SELECT * FROM spreads WHERE source='kiwoom' AND code=? ORDER BY ts`.

### FR-08 관측 전용 보장 (안전장치)

- 주문 엔드포인트(`kt10000` 계열) 호출 코드가 존재하지 않아야 하며, HTTP 클라이언트 화이트리스트는 `oauth2/token`만 허용한다.
- README 및 시작 로그에 “OBSERVATION MODE — NO ORDERS”를 명시한다.

### FR-09 실시간 관측 대시보드 (trakit 통합)

- trakit 프론트엔드에 **`#vi-arb` 메뉴(⚡ VI 차익)** 를 추가하고, 백엔드와 **WebSocket** 으로 실시간 스트림한다.
  - 경로: `GET(ws) /api/ws/vi-arb` (FastAPI). Vite proxy `ws:true`, nginx `Upgrade`/`map $connection_upgrade` 통과.
  - 메시지 스키마(JSON): `hello`(mode·비용파라미터), `vi`(발동/해제·`gubun`·`direction`·`vi_pct`·trigger_price), `spread`(`sec_since_vi`·krx_expected·nxt_best_ask·nxt_ask_qty·spread·cost·net_spread·`net_pct`·`first_buy`·`max_profit_pct`·opportunity), `krx_resume`(`randomend_sec`).
- **데이터 소스 2단 분리** (구현 빠르게 진행 + Kiwoom 검증 블로커 회피):
  - **A단계(완료)**: `services/vi_arb.py` mock 제너레이터 — 랜덤엔드 갭 시뮬. KRX VI 2단계 발동률(동적 3%/정적 10%) 반영.
  - **B단계(예정)**: mock 자리에 실제 Kiwoom WS observer(FR-02~06) 연결. **FE·메시지 스키마는 동일 유지** → 데이터만 교체.
- 대시보드 표시:
  - 연결상태(자동 재접속) · 통계(VI발동/기회/틱).
  - **방향 선택기**: `[전체] [▲ 상방 VI] [▼ 하방 VI]` — `direction(+/-)` 필터.
  - **기회 포착 테이블**: 종목·VI후(s)·KRX예상·NXT매도·잔량·스프레드·**순이익(원+%)**·**최대수익%**(최초매수가 기준 누적 최고).
  - **VI 이벤트 로그**: 발동/해제 단계 배지(정적/동적)+발동률, KRX 랜덤엔드 재개초.
- 관측 전용 — 대시보드에서 주문 트리거 UI 없음(FR-08 유지).

## 5. 외부 인터페이스 명세

### 5.1 키움 REST / WebSocket

- REST 토큰: `POST /oauth2/token` (운영 `https://api.kiwoom.com`)
- WebSocket: `wss://api.kiwoom.com:10000/api/dostk/websocket`
- 프레임: `LOGIN` / `PING`(echo) / `REG` / `REMOVE` / `REAL`
- 실시간 타입: `0B` 주식체결, `0D` 주식호가잔량, `1h` VI발동/해제
- NXT 시세 심볼: `{종목코드}_NX`, 통합호가: `{종목코드}_AL`
- 주요 FID: `10` 현재가/예상체결가, `20` 체결시간, `27` 최우선매도호가, `28` 최우선매수호가, `9001` 종목코드

### 5.2 ✅ 명세 검증 완료 (docs/kiwoom-rest-api.pdf)

| 항목 | 검증 결과 (페이지) |
|------|------|
| 실시간 등록 프레임 | `{"trnm":"REG","grp_no","refresh"(0해지/1유지),"data":[{"item":[],"type":[]}]}` (p.470) |
| 예상체결 타입 | **`0H` 주식예상체결** (p.504). 추정 `0E`는 시간외호가였음 — 정정 |
| VI(1h) FID | 9001 종목코드·302 종목명·**1221 발동가격**·**1225 적용구분(정적/동적/동적+정적)**·1224 해제시각·**1489 발동등락율(방향·발동률)**·9069 발동방향 (p.529) |

| NXT 심볼 | `{code}_NX`(NXT) / `{code}_AL`(SOR) (p.528) |

## 7. 운영 및 분석 (Phase 1 산출물)

- **일일 리포트(배치)**: VI 이벤트 수, 종목별 max(net_spread), opportunity 발생 횟수, 랜덤엔드 분포 히스토그램, 기회 시점의 NXT 잔량 분포.
- **Go/No-Go 판단 지표** (Phase 2 진입 조건 예시):
  - 주당 opportunity ≥ 10건, 기회당 중위 net_spread ≥ 비용의 2배,
  - 기회 시점 NXT 잔량 중위값 × net_spread 기준 기대수익이 인프라·운영 비용을 상회.
